# Команды боту (пульс / куки) — готчи, найденные 18 июля 2026

Реальные причины, доказанные по логам прогонов (`GET /executions/{id}?includeData=true`),
а не по словам бота. Записано, чтобы следующая сессия не повторяла возню.

## Готча №1 (самая дорогая): Telegram Trigger с двумя main-выходами НЕ роутит

Архитектура, которая НЕ сработала:
```
Мозг · триггер TG  --main[0]-->  Куки · это куки? (IF)
                     --main[1]-->  Пульс · команда? (IF)
```
Прогон 6929 (владелец написал `пульс`):
- В `runData` присутствуют: Мозг·триггер TG → Куки·это куки? → Мозг·проверка черновика
  → Мозг·есть черновик и ok? → Мозг·память 20 → Мозг·модель → Мозг·диалог → Мозг·ответ
- Узла **`Пульс · команда?` ВООБЩЕ НЕТ в runData** → ветка `[1]` никогда не запустилась.

Вывод: n8n source-node (Telegram Trigger) с двумя `main`-выходами пускает
только `[0]`. Командный роутинг одним триггером с ветками — НЕ делать.

### Что делать вместо этого (два рабочих пути)
- **(а) ОДИН IF-роутер сразу после триггера.** Триггер → `[0]` в единый IF
  («роутер команд»), который уже внутри ветвит: куки / пульс / всё остальное→агент.
  НЕ вешать две ветки прямо на source-node.
- **(б) ОТДЕЛЬНЫЙ Telegram Trigger с фильтром сообщений** (напр. точное слово
  `пульс` или `/пульс`) → прямо в цепочку Пульс (куки→снять→запись→ответ TG).
  Фильтр на уровне триггера надёжнее любого IF. Второй триггер перебивает вебхук
  бота только если у него тот же chat — делать отдельный командный триггер с фильтром
  по тексту, это безопасно.

## Готча №2: n8n-выражение `{{ $json.x or '' }}` → `invalid syntax`

Узел `Пульс · ответ TG` упал с `invalid syntax` из-за `{{ $json.hour or '' }}`.
Оператор `or` внутри `{{ }}` n8n недопустим.
- ✅ `{{ $json.hour }}`
- ✅ `{{ $json.hour ?? '' }}`  (null-coalescing — работает)
- ❌ `{{ $json.hour or '' }}`

## Готча №3: патч system-промпта агента НЕДОСТАТОЧЕН

После ДВУХ правок промпта `Мозг · диалог` (убрали «браузер не активирован») бот
ВСЁ РАВНО повторял эту ложь. Причина: sliding-память агента (20 сообщений)
тащит старые реплики, и агент «вспоминает» ложь из буфера, игнорируя новый промпт.
Браузер (browserless) при этом РАБОТАЛ — реальная причина молчания была мёртвый куки.

⇒ Детерминированные команды (пульс / куки) НЕ пропускать через LLM-агента.
  Роутинг = IF → прямой узел (HTTP/Code) → Telegram-ответ. Агент их вообще не видит,
  значит не может соврать. Это и есть рабочая архитектура v14 после 18 июля.

## Готча №4: токен куки брать ТОЛЬКО из панели, ПОСЛЕ капчи+логина

Прогон 6886: первый токен владельца (`lmnaqabr…`) → `{"alive":false,"note":"редирект на логин"}`.
Причина: снят ДО входа в панель (на login.uptaxi.ru). Кука `cabinetUpTaxi` появляется
только ПОСЛЕ успешного логина с капчей.
- ✅ Зайти в `https://1165.upphone.ru:4422/web/cab/1165/`, дождаться меню,
  ТОЛЬКО ТОГДА F12 → Console →
  `document.cookie.split('; ').find(c => c.startsWith('cabinetUpTaxi=')).split('=')[1]`
- ❌ Брать на `login.uptaxi.ru` — токен пустой/мёртвый, UpTaxi отвергнет.

«Всеядная» ветка куки (regex `куки|[a-zA-Z0-9]{16,64}`) ловит и `куки <токен>`,
и голый токен — подтверждено прогоном 6910 (`alive:true, na_linii:7, gorit:1, available:2`).
Шли боту любым из вариантов.

## Готча №5: цифры Пульса могут не совпадать с тем, что видит владелец

Прогон 6915 вернул `na_linii:7`, а владелец в панели видел 18. Две причины:
- живые данные меняются каждую минуту (водители выходят/заходят);
- бот читает колонку `na_linii` грида, владелец смотрит, возможно, на другую.
⇒ Если расхождение — не гадать, попросить владельца ткнуть в нужную колонку
  (какие поля есть в гриде, видно в коде узла `Пульс · снять`).
